Getting started

Learn more »

Step 1: Get these two scripts. 

curl -s -O https://chronica.co/sample/custom.chronica.sh
curl -s -O https://chronica.co/sample/chronica.sh

Step 2: Edit custom.chronica.sh (for you and yours) 

$ vi custom.chronica.sh
subscriptions="you@your.org"
admins="you@your.org"
orgDomain="your.org"

Step 3. Review chronica.sh (so you know it's cool)

Step 4: Test 

sh chronica.sh

Step 5: Run 

sh chronica.sh run

Step 6: (to https://chronica.co using Mozilla Persona)

Click on alert rows to see their content (except on https://chronica.co/demo where you can preview the admin's alerts.)

ps. The script posts to https://secure.chronica.co, which you can check as follows, i.e. that it is allowed by your firewall.

$ sh chronica.sh https secure.chronica.co 443
OK - chronica.co port 443 https available (HTTP/1.1 200 OK)

Getting jiggy with it

Customize the custom.chronica.sh script to check the ports and services you wish to monitor, every minute, hour or daily.

c0hourly() {
  c1topic hourly
  c0diskspace
  c2rhttps bizswitch.net 443 # request external check by chronica server
  c2notcp chronica.co 21
  c2tcp chronica.co 80
  c2https chronica.co 443
}

where the convention for naming functions indicates the number of parameters. Commands can be tested from the command-line, as follows.

$ sh chronica.sh nohttp chronica.co 80
OK - chronica.co (port 80) http unavailable (HTTP/1.1 301 Moved Permanently)

Add Nagios plugins and adhoc scripts as required.

c0minutely() {
  c1topic minutely
  c0load
  c2httpsAuth secure.chronica.co 443 
  /usr/lib/nagios/plugins/check_pgsql -H localhost 
  ssh postgres@localhost scripts/check_replication.sh
}

We suggest creating functions for your custom checks.

c0pgsql() { 
  /usr/lib/nagios/plugins/check_pgsql -H localhost &&
    ssh postgres@localhost scripts/check_replication.sh
}

Then you can test such functions individually from the command-line as follows.

$ sh scripts/chronica.sh pgsql
CRITICAL - no connection

As good practice, run chronica.sh as an unpriviledged user e.g. from rc.local using su. Use ssh forced-commands and/or sudo to grant only those priviledges actually required for running your custom checks.

$ cat /etc/rc.local
su - chronica -c 'scripts/chronica.sh start'

News

2014-01-02   Hourly charts added
The maximum and average of minutely metric series are recorded for hourly charts. Metric series are factored into a byte array to reduce memory requirements to 7k metrics per megabyte.

2013-12-28   Charts added (minutely)
Support implemented for metrics (values) which are graphed using Chartjs.org.

2013-12-15   Info sections added
"Getting Started," "Frequently Asked Questions" and "News" sections have been added to the site. When signed in, click on "Chronica.co" for FAQ and News, and click on "Help" to see the "Getting Started" info. (A more detailed "User Guide" is still required.)

2013-12-01   AngujarJS UI started
AngularJS is great! Goodbye formerly beloved jQuery. (See app.js and app.html.)

2013-11-24   Project inception

Taking the all-important third crack at a monitoring server, following two previous monitoring projects over the past 18 months, namely Bizstat and Croc (on http://github.com/evanx/vellumapp).

Bizstat is a single-tenancy internal active server, geared for custom checks, but with general checks thrown in, via plugin shell scripts, which would use NRPE (Nagios remote plugin executor) or ssh forced commands for remote active checks. This project was abandoned in favour of pursing of multi-tenancy cloud-based approach.

Croc ("Cron Output Collector") is a multi-tenancy passive server, and the inspiration for Chronica. Bootstrap 2 and jQuery as used for its UI, and JDBC for persistence. Prototyping was tedious, where UI changes required Java servlet changes which required SQL fixes, which diminished the speed and enjoyment of UI prototyping. Clearly a better approach for rapid prototyping was needed e.g. JPA for back-end persistence, and a Javascript data-binding framework for the front-end.

The result is Chronica, which will use the AngularJS front-end framework, and JPA.

Frequently asked questions

What's this?
A cloud-computing monitoring service.

What does it compute for me?
Information you submit that you want to monitor, e.g. scheduled checks. As a default criterion, our server checks if the information has changed, and if so, will alert you. If the messages stop, we will alert you that something might be wrong. If it is changing all the time, we will ignore it for you, since alerts are not useful in that case.

What security?
SSL certificates authenticate publishers, and Mozilla Persona authenticates subscriptions.

How much does it cost?
It's free and open. Bits and bytes are priceless ;)

What use cases?
Use Chronica to monitor your networks and services, internally and externally, to ensure performance and security. Add custom checks to monitor your specific infrastructure and applications. When something is wrong, be the first to know :)

Who should try it?
At this stage, anyone who wants monitoring, without any server setup or customistion required, up and running in a minute. Also, Linux admins who want a monitoring solution that enables custom checks using shell commands.

How is this different from some other cloud-based monitoring services?
It's simple yet powerful for custom monitoring. It's opensource, so you can own it.

How is this different from some other opensource monitoring projects?
We provide a free multi-tenancy cloud service for instant monitoring without any server setup. Our solution supports Nagios plugins, and will be integrated with opensource agents, according to demand.

What notification mechanisms are supported?
Primarily, email. Also "push" notification via HTTPS post to a URL you can register. (Naturally this URL should require client auth and trust our SSL cert, and allow our IP number to connect.) This could provide an SMS gateway on your account. 

$ sh chronica.sh push https://myserver:8443/myapp

What about mobile push notifications?
We want to support mobile push notifications, perhaps via our own app e.g. using PhoneGap aka Apache Cordova with its PushPlugin. Also Facebook, Google+ and Twitter integration for messaging via those platforms. Perhaps even a Jabber server to which you hook up a mobile IM app e.g. Xabber.

What other notification mechanisms are already supported?
You can poll our server via HTTPS for your alerts. 

$ sh chronica.sh poll
Time: 16:02:54
Subject: root@hostx checksys CRITICAL

What technologies are used?
AngularJS for the front-end, making JSON requests to a Java back-end. We use the JRE's built-in HTTPS server rather than Apache Tomcat. Mozilla Persona is used for authentication.

Why doesn't the site work on my phone's browser?
It will, soon! We use Twitter Bootstrap 3, which is "mobile first" and responsive to desktop, tablet and phone.

Windows or Linux?
We provide a "starter" script for Linux, which can be used out the box to actively monitor Windows machines remotely via TCP. We will integrate an opensource Windows agent, such as the Nagios Cross-Platform Agent (NCPA), and/or our own Chronica Java-based agent.

What can be installed on Linux clients right now?
A shell script that uses curl to post status information to the service via HTTPS. Out of the box, this client script uses standard diagnostic tools such as df, ping, netcat and openssl. However, the system is specially engineered to support Nagios plugins, and custom checks. 

curl -s https://chronica.co/sample/chronica.sh -o chronica.sh

How is it run?
The script can be run as a background process, to run minutely, hourly and daily checks, without needing crond. (It sleeps in a loop.)
Also, you can pipe the output of custom cron jobs to this script.
Finally, the script can be used as your MTA (via the /etc/alternatives/mta symlink) to handle all cron job output, i.e. if your server only uses its MTA for cron emails. 

$ sh chronica.sh start

How is it updated?
The script itself will provide hints, to verify and update. 

$ sh chronica.sh update
curl -s https://chronica.co/sample/chronica.sh | sha1sum
Verifying signature using ~/.chronica/etc/chronica.pub.pem
OK: verified: https://chronica.co/sample/chronica.sh.sha1.sig.txt

What API does it use?
Text document with email-style headers, submitted via HTTPS POST. (HTTP headers are parsed as well). This means we can pipe the output of a shell script to curl, to post to https://secure.chronica.co. The content typically contains headers such as "Topic" and "Subscribe." Some basic HTML tags are acceptable to improve rendering e.g. headings, bold, italics. But the text is assumed to be preformatted with line-breaks, and so it is not strictly an HTML document. 

$ echo "OK - <i>nothing wrong</i>" | sh chronica.sh postheaders \
  'Topic: System Health' 'Alert: STATUS_CHANGED' 'Subscribe: me@mine.org'

How are checks performed?
Your scripts can execute Nagios plugins, shell commands, and/or other programs, to produce the output that you wish to monitor.
For example, commands might list the hashes of sensitive files, verify installed packages, grep log files for failed logins, etc.
So it's totally customisable. As such, we will extend our script soon to satisfy PCI DSS compliancy, out of the box :)

What about checking my site from the outside?
Naturally you can use Chronica from a server in a different data center, or even your home computer, to monitor your site externally. But the easiest is to request the Chronica server to perform external checks for you. 

$ echo "Check-https: www.google.com 443" | sh chronica.sh post
OK: www.google.com port 443 https: CN=www.google.com,O=Google Inc,L=Mountain View,ST=California,C=US

How are client certs managed?
The client shell script automatically generates SSL certs according to your organisation's domain, as configured in your custom script. By default the user and hostname is used for the cert's common name. It enrolls admins you similarly configure. As the first admin enrolled for your organisation, you are automatically approved. Admins can confirm client certs and approve other admins, via the site.

How are subcribers managed?
The message content specifies its topic and invitees, via email-style headers. If you have approved the topic's cert, then this topic will be published. However, invitees must themselves subscribe to specific topics. This is performed via the site, which validates their email address using Mozilla Persona. So you should advise them visit the site to subscribe to available topics.

How do subcribers opt-out?
Actually they have to opt-in themselves, and can unsubscribe and re-subscribe anytime.

How are alerts triggered?
Typically by any change in the monitored output content. However, the content is specially processed to ignore suprious information, to avoid unnecessary alerts. Otherwise just grep and sed out the salient information that you wish to monitor for changes. And send us any suggestions for improvements e.g. how to better filter content. Code is even better.

What other applications?
We are looking at using the platform for scheduling reminders.
