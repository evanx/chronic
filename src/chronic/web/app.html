Getting started

Learn more »

Step 1: Get two scripts, as an unpriviledged user. 

wget https://chronica.co/sample/custom.chronica.sh
wget https://chronica.co/sample/chronica.sh

Step 2: Edit custom.chronica.sh (for you and yours) 

$ vi custom.chronica.sh
subscribers="you@your.org"
admins="you@your.org"
orgDomain="your.org"

Step 3. Review chronica.sh (so you know it's cool)

Step 4: Test 

sh chronica.sh

Step 5: Run 

sh chronica.sh run

Step 6: (to https://chronica.co using Mozilla Persona)

Click on alert rows to see their content (except on https://chronica.co/demo where you can preview my sample alerts.)

ps. The script posts to https://secure.chronica.co.

$ sh chronica.sh https chronica.co 443
OK - chronica.co port 443 https available (HTTP/1.1 200 OK)

Getting jiggy with it

Customize the custom.chronica.sh script to check the ports and services you wish to monitor, every minute, hour or daily.

c0hourly() {
  c1topic hourly
  c0diskspace
  c2notcp chronica.co 21
  c2tcp chronica.co 80
  c2https chronica.co 443
}

where the convention for naming functions indicates the number of parameters. Commands can be tested from the command-line, as follows.

$ sh chronica.sh nohttp chronica.co 80
OK - chronica.co (port 80) http unavailable (HTTP/1.1 301 Moved Permanently)

Add Nagios plugins and adhoc scripts as required.

c0minutely() {
  c1topic minutely
  c0load
  c2httpsAuth secure.chronica.co 443 
  /usr/lib/nagios/plugins/check_pgsql -H localhost 
  ssh postgres@localhost scripts/check_replication.sh
}

We suggest creating functions for your custom checks.

c0pgsql() { 
  /usr/lib/nagios/plugins/check_pgsql -H localhost &&
    ssh postgres@localhost scripts/check_replication.sh
}

Then you can test such functions individually from the command-line as follows.

$ sh scripts/chronica.sh pgsql
CRITICAL - no connection

As good practice, run chronica.sh as an unpriviledged user e.g. from rc.local using su. Use ssh forced-commands and/or sudo to grant only those priviledges actually required for running your custom checks.

$ cat /etc/rc.local
su - chronica -c 'scripts/chronica.sh start'

Instant cloud monitoring

Frequently asked questions

What's this?
A cloud-computing monitoring service.

What does it compute in the cloud?
Information from computers you want to monitor. Our server checks if the information has changed, and if so, will alert you. If it is changing all the time, we will ignore it for you. If the messages stop, we will alert you that something might be wrong.

What is it essentially?
Similar to email, except that "messages" are submitted using client-authenticated HTTPS, and "recipients" must subscribe to recieve alerts when the message changes e.g. "your website was up but now it is down."

How much does it cost?
It's free and open. Bits and bytes are priceless ;)

What is its value proposition?
Any text output you can submit via HTTPS, you can monitor - instantly and freely :)

What use cases?
We suggest you use Chronica to monitor your networks and applications, to ensure your IT performance and security. When something is wrong, be the first to know :)

How is this different from some other cloud-based monitoring services?
It's simple yet powerful for custom monitoring. It's opensource, so you can own it.

How is this different from some other opensource monitoring projects?
We provide a free multi-tenancy cloud service for instant monitoring without any server setup. Our solution supports Nagios plugins, and will be integrated with opensource agents, according to demand.

What notification mechanisms are supported?
Only email. Support is envisaged for mobile push notifications, but not SMS. (Currently email is disabled for alpha testing.)

What technologies are used?
AngularJS for the front-end, making JSON requests to a Java back-end. We use the JRE's built-in HTTPS server rather than Apache Tomcat. Mozilla Persona is used for authentication.

Why doesn't the site work on my phone's browser?
It will, soon! We use Twitter Bootstrap 3, which is "mobile first" and responsive to desktop, tablet and phone.

Windows or Linux?
We provide a "starter" script for Linux, which can be used out the box to actively monitor Windows machines remotely via TCP. However, you can expect integration with an opensource Windows agent in coming months, such as the Nagios Cross-Platform Agent (NCPA).

What can be installed on Linux clients right now?
A shell script that uses curl to post status information to the service via HTTPS. Out of the box, the client uses standard diagnostic tools such as df, ping, netcat and openssl. However, the system is specially engineered to support Nagios plugins, and custom checks.

How is it run?
The script can be started, to run in the background. It has built-in support for minutely, hourly and daily checks, without needing crond. (It has a loop but sleeps most of the time.) Also, you can pipe the output of cron jobs to our script. Incidently, the script can be used as your MTA (via the /etc/alternatives/mta symlink) to handle all cron job output, i.e. if your server only uses its MTA for cron emails.

What API does it use?
Text document with email-style headers, submitted via HTTPS POST. For example, our shell script pipes to curl to post to https://secure.chronica.co. Some basic HTML tags are acceptable to improve rendering e.g. headings, bold, italics. But the text is considered preformatted with line-breaks, and so is not strictly an HTML document.

How are checks performed?
Your scripts can execute Nagios plugins, shell commands, and/or other programs, to produce the output that you wish to monitor. For example, commands might list the hashes of sensitive files, verify installed packages, grep log files for failed logins, etc. So it's totally customisable. As such, we will extend our script soon to satisfy PCI DSS compliancy, out of the box :)

How are client certs managed?
The client shell script automatically generates SSL certs according to your organisation's domain, as configured in your custom script. By default the hostname of your machine is used for the cert's common name. It enrolls admins you similarly configure. As the first admin enrolled for your organisation, you are automatically approved, and can confirm client certs, and grant admin priviledges to others, via the site.

How are subcribers managed?
The message content specifies its topic, and invites subscribers by their email addresses, via email-style headers, namely "Topic:" and "Subscribers:". If you have approved the topic's cert, then this topic will be published. However, invitees must themselves subscribe to specific topics. This is performed via the site, which validates their email address using Mozilla Persona. So you should advise them visit the site to subscribe to available topics. This way we don't send unsolicited email to unauthentified addresses.

How do subcribers opt-out?
Actually they have to opt-in themselves, and can unsubscribe and re-subscribe anytime.

How are alerts triggered?
Typically by any change in the monitored output content. However, the content is specially processed to ignore suprious information, to avoid unnecessary alerts. Otherwise just grep and sed out the salient information that you wish to monitor for changes. And send us any suggestions for improvements e.g. how to better filter content. Code is even better.

How do I to contribute my improvements for inclusion?
Fork https://github.com/evanx/chronic. See chronica.sh and StatusRecordMatcher. Email us about your ASL-licensed shell functions and Java text processing classes.

Show me the code!
See https://github.com/evanx/chronic. This uses https://github.com/evanx/vellumcore which is a common base of distilled code from previous projects.

2013-11-24   Project inception

Taking the all-important third crack at a monitoring server, following two previous monitoring projects over the past 18 months, namely Bizstat and Croc (on http://github.com/evanx/vellumapp). Chronica's mantra is "wonderfully simple, incredibly powerful."

Bizstat is a single-tenancy internal active server, geared for custom checks, but with general checks thrown in, via plugin shell scripts, which would use NRPE (Nagios remote plugin executor) or ssh forced commands for remote active checks.

Croc ("Cron Output Collector") is a multi-tenancy passive server, and the inspiration for Chronica. Its UI uses Bootstrap 2 and jQuery, and it used JDBC from word go, albeit using an H2 in-memory database. Prototyping was tedious, where UI changes required Java servlet changes which required SQL fixes, which diminished the speed and enjoyment of UI prototyping. Clearly a better approach for rapid prototyping was needed.

The result is Chronica, which uses the awesome AngularJS front-end framework (see app.js and app.html), and a pluggable SQL'esque storage engine mechanism (see EntityStore and MapStore). Prototyping can forgo SQL, using "key" objects to support queries. See ChronicMapStore and ChronicStorage. A JDBC implementation can enable persistence later, without changing the application. See CertStore.
